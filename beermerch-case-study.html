Overview
Context

Telegram Mini Apps are a relatively new frontier for commerce. They run inside the Telegram messenger, which means users never leave the app — the entire shopping experience, from browsing merch to paying, happens in a familiar environment.

A real craft beer brand wanted to sell branded merchandise through a Telegram Mini App. They had an audience, a brand identity, and a product. They needed a complete, production-ready UX and UI.

Why Telegram Mini Apps are interesting to design for
Mini Apps live inside a constrained environment. You inherit Telegram's UI patterns (the bottom sheet, the main button, the header) but still need to design a branded, coherent experience within those constraints. It's a different challenge from open-web design — you're designing within a system, not on a blank canvas.
The Task

Design the full UX and UI of the merch store: from first open to order confirmed. This included not just the customer-facing app but also an admin panel for order management.

Design Challenges
01
Platform constraints
Telegram Mini Apps operate within the messenger's chrome. Navigation, back buttons, and the main action button are all partially controlled by Telegram. Designing within this without losing brand identity was the core constraint.
02
Complete checkout flow
A real purchase flow requires multiple delivery options, address entry, payment, and confirmation — all on mobile, all within the Mini App. Every step needed to be frictionless. Drop-off at any point means a lost sale.
03
Edge cases & error states
A production-ready design can't ignore what happens when things go wrong. Empty basket, 404, 505 server errors — every failure state was designed so the user always knows what happened and has a clear path forward.
Process
01
Platform Research
Studied Telegram Mini App guidelines, constraints, and existing examples to understand the design system I was working within.
02
Flow Mapping
Mapped every possible user path — from first open through each delivery option, payment method, and error state.
03
Component System
Built a design system in Figma: typography, color tokens, icon set, and a full component library before touching screens.
04
Screen Design
Designed all screens including all states — loading, empty, error — to ensure the app is production-complete, not just happy-path.
05
Admin Panel
Designed an admin interface for order management, extending the same visual system so both customer and admin sides feel unified.

Key Decisions
01
Why dark theme throughout?
The brand aesthetic and the Telegram night mode context both pushed toward dark. More importantly, dark UI with a bold yellow accent creates a distinctive identity — this app looks like nothing else in a user's Telegram.
02
Why two separate delivery flows (self-pickup + CDEK)?
Combining them into one form would create conditional logic that confuses users. Separating them into distinct, focused screens means each flow only shows what's relevant. Fewer fields visible = less anxiety at checkout.
03
Why design error states and edge cases in detail?
This is a real product going live. A beautiful happy path that breaks on errors is a bad product. Users who hit a 404 with no recovery path abandon immediately. Every edge case designed = fewer lost customers.
04
Why include an admin panel in scope?
The shop is useless without someone managing orders. Including the admin panel in the design scope ensures the system is thought of holistically — not just the customer experience, but the operational one too.
Reflection
What worked
Thinking in systems
Building the component library first — typography tokens, color variables, icon set — meant scaling to 15+ screens was methodical, not chaotic. Every new screen was assembly, not invention from scratch.
What worked
Designing for the real context
Telegram Mini Apps aren't just "mobile apps in a box." Users are in their messenger, probably mid-conversation. Decisions about loading states, transitions, and the persistent main button were all informed by that context.
What I'd do differently
User testing before handoff
Given the tight scope, I didn't run any usability testing on the checkout flow. The delivery method split felt logical to me as a designer, but I'd want to validate it with actual users to confirm it reduces friction rather than adding a step.
What I'd do differently
Onboarding flow
The app opens directly to a homepage. For a brand's first-time visitors, a brief onboarding moment — what this is, why it's worth exploring — could improve initial engagement. I'd explore that in a next iteration.
